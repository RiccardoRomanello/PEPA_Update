Capacity planning

Real systems are expected to service some demand with good performance. For example, a cloud based web-service may have a Service Level Agreement (SLA). This SLA might stipulate that the web-service will have a number of clients when operating (service demand) and must respond to all client requests within a certain time (performance).

Real systems are usually complex systems, consisting of compositions of resources. A naive approach to satisfying the SLA would be to continue adding resources to the system until the system meets the performance requirement. We want to avoid over-provisioning, where we have under-used resources, because these add to our operational cost unnecessarily. We want to avoid under-provisioning, as this means our system is not going to meet the SLA.

We want an optimum configuration of resources, a configuration that uses the fewest number of resources, but still meets the expected service demand with a good performance. Searching for optimal configurations is time consuming and difficult. The CPT is an automated tool designed to aid the modeller in finding an optimal confguration, and in gaining some insight into the systems' behaviour when we consider the number of resources as a parameter.

This guide

This guide provides a step-by-step description of the Capacity Planning Tool (CPT).

It shows how to configure the tool in order to search for some optimal model configuration of system components. This page gives an overview of the tool in order to provide insight into how the tool works, and therefore how to configure the tool for a search. We start with a brief motivation, and then provide a high level view of the CPT.

Familiarity with PEPA is assumed. To learn more about PEPA visit the PEPA web site.

Tool overview

  1. The designer creates a PEPA model in the PEPA editor.
  2. Then the designer starts the CPT choosing either a Driven Particle Swarm Optimisation (dPSO) or a Single Particle Swarm Optimisation (PSO) search.
  3. The search engine then searches through the model configuration search space, using a cost function to evaluate each model configuration.
  4. The CPT passes a tree view of the results, with the model configuration with the least cost at the top.
  5. The CPT also passes a list view of the results, sorted in ascending cost.

Performance evaluation

The CPT can evaluate models in terms of:
  1. Capacity Utilisation
  2. Population levels
  3. Throughput
  4. Average Response-Time

The search algorithm (in brief)

The CPT uses Particle Swarm Optimisation (PSO) to search through a model configuration search space. The PSO creates a population of particles. This population then moves through the model configuration search space a number of times seeking an optimal configuration.

Particles have a value derived from the corresponding model configuration using a cost function. The parameters of the cost function will have been defined by the designer. Therefore the meaning of optimal, in terms of a model configuration, will be dependent on the user-defined cost function.

In a dPSO the CPT uses a Hill Climbing algorithm in order to find the best PSO settings for a model configuration search.
